## Commands

```sh
yarn install          # Install dependencies
yarn build            # Compile TypeScript → dist/ via esbuild (bundles + minifies)
yarn test             # Run unit + integration + e2e tests (requires Docker)
yarn test:coverage    # Run all tests with coverage report
yarn test:unit        # Unit tests only (no Docker required)
yarn test:integration # Integration tests in Docker containers (Node 14, 16, 18, 20)
yarn test:e2e         # E2E tests in Docker containers (runs yarn build first)
yarn test:smoke       # Smoke tests against the published npm package
yarn check:lint       # ESLint
yarn check:format     # Prettier
yarn check:types      # TypeScript type checking (no emit)
```

To run a single test file:

```sh
yarn test:unit --testPathPattern=git-interop
```

## Architecture

`branch-commit-msg` is a git `commit-msg` hook tool with two entry points that get bundled separately:

- **`src/index.ts`** → `dist/index.js` (the `branch-commit-msg` CLI): uses yargs to expose the `install` command, which copies the compiled hook binary into `.git/hooks/commit-msg`.
- **`src/commit-msg-hook.ts`** → `dist/commit-msg` (the hook itself): runs at commit time, reads `.commitmsgrc.json`, extracts the regex match from the active branch name, and rewrites the commit message file in-place.

Supporting modules:

- **`src/config.ts`**: Reads and validates `.commitmsgrc.json`. Returns `undefined` silently if missing/invalid (hook exits 0, no-op).
- **`src/git-interop.ts`**: Thin wrappers around `git symbolic-ref --short HEAD` and `git rev-parse --show-toplevel`. Both functions accept an injectable `git` function for unit testing (dependency injection pattern used to avoid spawning real git in unit tests).
- **`src/install.ts`**: Copies `dist/commit-msg` to `.git/hooks/commit-msg` of the target repo.

## Configuration

`.commitmsgrc.json` (placed at the repo root) controls hook behavior:

```json
{
  "extractPattern": "([A-Z]+-[0-9]+)",
  "extractPatternMatchCase": false,
  "commitMsgFormat": "[$1] $MSG"
}
```

- `extractPattern`: regex applied to the branch name; first capture group is extracted
- `extractPatternMatchCase`: whether the regex match is case-sensitive
- `commitMsgFormat`: template for the rewritten commit message (`$1` = capture group, `$MSG` = original message)

If the file is missing or invalid, the hook exits 0 (no-op).

## Test Structure

Tests are split across four suites with distinct purposes:

| Suite       | Location            | Docker | Notes                                                            |
| ----------- | ------------------- | ------ | ---------------------------------------------------------------- |
| Unit        | `test/unit/`        | No     | Mocks git via dependency injection                               |
| Integration | `test/integration/` | Yes    | Runs against real filesystem/git; tested across Node 14/16/18/20 |
| E2E         | `test/e2e/`         | Yes    | Requires built `dist/`; tests the full hook flow                 |
| Smoke       | `test/smoke/`       | No     | Tests the published npm package post-release                     |

`test/containerized-test.ts` is a utility that spins up Docker containers for integration/e2e suites. Each suite runs sequentially across all supported Node versions.

## Workflow

This project follows test-driven development (TDD). When implementing features or fixes:

1. Write a failing test first (`yarn test:unit` to confirm it fails)
2. Implement the minimum code to make the test pass
3. Refactor if needed, keeping tests green

## Pull Requests

PR titles must follow [Conventional Commits](https://www.conventionalcommits.org/) format and be written in the passive voice:

- `feat: branch name extraction is now configurable`
- `fix: config file is read from repo root, not cwd`
- `chore: dependencies are bumped to latest`

## Build Output

The build produces two self-contained bundled binaries in `dist/` (no `node_modules` needed at runtime):

- `dist/index.js` — the CLI
- `dist/commit-msg` — the hook script

Only the `dist/` directory is published to npm (see `"files"` in `package.json`).

> **Do not edit files in `dist/` directly.** They are generated by `yarn build` and any manual changes will be overwritten. Make changes in `src/` instead.
